01The Problem

Reputation is broken

Every signal used to evaluate developers today is either manipulable, subjective, or irrelevant to engineering ability.

⭐
GitHub Stars
Stars measure marketing, not engineering. A viral README with zero meaningful code can accumulate tens of thousands of stars. They say nothing about a developer's actual ability to build production systems.
📱
Social Followers
Twitter/X, LinkedIn, and blog followings reward content creators, not code writers. The most prolific engineers are often the most invisible socially. Follower counts are gameable and decay rapidly.
📄
Resumes & Interviews
Self-reported skills and whiteboard performance correlate weakly with real-world engineering output. The best engineers often perform worst in traditional interview formats.
⚙️
What POW Measures
Raw commit velocity. Project complexity depth. Contribution consistency over years. Real repository impact. Trajectory acceleration. All from publicly visible, tamper-evident GitHub activity — the actual record of work.
02The Solution

The execution index

Code is one of the few professions where your complete work history is publicly visible. We use it.

POW is a developer performance intelligence platform that ingests public GitHub activity across millions of commits to produce a single, continuously updated Execution Score per developer.

Unlike static profiles or self-reported credentials, the score is derived entirely from observable engineering behavior — what you commit, to what, how often, and how complex.

The result is a global leaderboard ranked by real engineering output, filterable by country, city, and technology stack.

03Scoring Engine

How the score is computed

The scoring engine processes raw GitHub data through a five-component weighted model to produce a normalized 0–10,000 execution score. Each component draws from different data subsets. The components are normalized independently against the global developer population using PostgreSQL window functions, ensuring scores reflect percentile standing rather than absolute numbers.

scoring-engine/calculate.ts
interface ScoringInput {
  commits:      Commit[];
  repositories: Repository[];
  windowDays:   90 | 180 | 365;
}

async function calculateExecutionScore(
  input: ScoringInput
): Promise<ExecutionScore> {

  const velocity    = await scoreVelocity(input.commits, input.windowDays);
  const complexity  = await scoreComplexity(input.repositories);
  const consistency = await scoreConsistency(input.commits);
  const impact      = await scoreImpact(input.repositories);
  const trajectory  = await scoreTrajectory(input.commits, velocity);

  // Normalize each component against global population
  const [nV, nC, nK, nI, nT] = await Promise.all([
    normalize(velocity,    'velocity'),
    normalize(complexity,  'complexity'),
    normalize(consistency, 'consistency'),
    normalize(impact,      'impact'),
    normalize(trajectory,  'trajectory'),
  ]);

  // Weighted sum → 0 to 10,000
  const raw = (nV*.20) + (nC*.25) + (nK*.20)
            + (nI*.25) + (nT*.10);

  return {
    executionScore: Math.round(raw * 10000),
    computedAt:     new Date(),
  };
}

Score update cadence: Scores are recomputed whenever the ingestion pipeline detects new commit activity. High-activity developers get near-real-time updates via priority queue.

Normalization strategy: Each sub-score is normalized using a percentile rank against the full developer population using PostgreSQL window functions.

normalize.sql
-- Percentile rank for velocity
SELECT
  developer_id,
  PERCENT_RANK() OVER (
    ORDER BY velocity_raw ASC
  ) AS velocity_pct
FROM developer_scores
WHERE is_bot = FALSE
  AND gaming_flag = FALSE;

-- Update scores atomically
UPDATE developers
SET
  execution_score = :score,
  global_rank = RANK() OVER (
    ORDER BY execution_score DESC
  ),
  score_updated_at = NOW()
WHERE id = :developer_id;

06Data Ingestion Pipeline

GitHub ingestion pipeline

Fast, resilient, and immune to GitHub's rate limiting constraints.

ingestion/sync-developer.ts
export async function syncDeveloper(
  username: string
): Promise<SyncResult> {

  // 1. Fetch GitHub user profile
  const profile = await github.getUser(username);
  await db.developers.upsert({ ...profile });

  // 2. Fetch all repositories (paginated)
  const repos = await github.getUserRepos(username, {
    type: 'owner', per_page: 100,
  });

  // 3. Queue per-repo commit harvesting
  for (const repo of repos) {
    await commitQueue.add('harvest', {
      repoId: repo.id,
      since:  repo.lastSyncedAt ?? null,
    }, { priority: getPriority(repo.stars) });
  }

  // 4. Fetch language breakdown (GraphQL)
  const langs = await github.getLanguages(repos);
  await db.techStacks.upsert(profile.id, langs);

  // 5. Queue score recalculation
  await scoreQueue.add('recalculate', {
    developerId: profile.id,
  }, { delay: 5000 });

  return { status: 'queued', repos: repos.length };
}

Delta sync strategy: Only re-fetch commits newer than the last sync timestamp. GitHub ETags stored per-repository reduce API calls by up to 80%.

Token rotation: Multiple GitHub API tokens rotated round-robin. At 10 tokens, that's 50,000 requests/hour.

Queue priorities: New submissions get priority-1 (immediate). Daily delta syncs get priority-2. Weekly full syncs get priority-5.

07Anti-Gaming System

Making it tamper-resistant

Eight detection signals evaluated per-commit and per-developer to filter non-genuine engineering activity.

🚨 Attack Vector
Empty Commit Spam
Hundreds of trivial commits (0 additions, 0 deletions) to inflate velocity scores. Often scripted via bots with cronjobs.
✓ Detection Method
Minimum Delta Threshold
Commits below a minimum viable delta (additions + deletions < 5) are flagged with gaming_score > 70 and excluded from velocity calculations.
🚨 Attack Vector
Repetitive File Cycling
Repeatedly modifying the same 1–2 files (e.g. a README counter) to simulate active development without real code changes.
✓ Detection Method
File Entropy Analysis
The ratio of unique files touched vs total commit count is tracked. Entropy below a threshold triggers a gaming_flag.
🚨 Attack Vector
Bot Account Activity
Using a bot GitHub account to make commits, or running automated push scripts that generate fake commit histories.
✓ Detection Method
Commit Pattern Analysis
Commits clustered at identical time intervals are flagged. Human patterns have natural variance. Perfect regularity is a bot signal.
🚨 Attack Vector
Star Farming
Creating star-exchange networks where developers collectively star each other's repos to inflate the impact signal.
✓ Detection Method
Star Velocity Anomaly
Stars from accounts with low follower counts or accounts created within 30 days of the starring event are weighted down.
