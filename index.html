01 The Problem

Reputation is broken

Every signal used to evaluate developers today is either manipulable, subjective, or simply irrelevant to engineering ability.

⭐
GitHub Stars
Stars measure marketing, not engineering. A viral README with zero meaningful code can accumulate tens of thousands of stars. They say nothing about a developer's actual ability to build production systems.
📱
Social Followers
Twitter/X, LinkedIn, and blog followings reward content creators, not code writers. The most prolific engineers are often the most invisible socially. Follower counts are gameable and decay rapidly.
📄
Resumes & Interviews
Self-reported skills and whiteboard performance under artificial pressure correlate weakly with real-world engineering output. The best engineers often perform worst in traditional interview formats.
⚙️
What POW Measures
Raw commit velocity. Project complexity depth. Contribution consistency over years. Real repository impact. Trajectory acceleration. All derived from publicly visible, tamper-evident GitHub activity — the actual record of work.
02 The Solution

The execution index

Code is one of the few professions where your complete work history is publicly visible. We use it.

POW is a developer performance intelligence platform that ingests public GitHub activity across millions of commits and thousands of repositories to produce a single, continuously updated Execution Score per developer.

Unlike static profiles or self-reported credentials, the execution score is derived entirely from observable engineering behavior — what you actually commit, to what, how often, and how complex.

The result is a global leaderboard ranked by real engineering output, filterable by country, city, and technology stack.

03 Scoring Engine

How the score is computed

The scoring engine is the analytical core of POW. After the ingestion pipeline collects raw GitHub data, the engine processes it through a five-component weighted model to produce a normalized 0–10,000 execution score.

Each component draws from different subsets of the raw data. Velocity pulls from commit timestamps and additions/deletions counts. Complexity draws from repository language breakdowns, file tree depth, and structural entropy. Consistency uses the statistical distribution of commit timestamps over long windows. Impact aggregates repository graph signals. Trajectory fits a curve to the velocity and complexity time series to calculate acceleration.

The components are normalized independently against the global developer population, then weighted and summed. This ensures that the score reflects percentile standing rather than absolute numbers, which vary wildly across developer types.

scoring-engine/calculate.ts 
interface ScoringInput {
  commits:      Commit[];
  repositories: Repository[];
  techStacks:   TechStack[];
  windowDays:   90 | 180 | 365;
}

async function calculateExecutionScore(
  input: ScoringInput
): Promise<ExecutionScore> {

  const velocity    = await scoreVelocity(input.commits, input.windowDays);
  const complexity  = await scoreComplexity(input.repositories, input.techStacks);
  const consistency = await scoreConsistency(input.commits);
  const impact      = await scoreImpact(input.repositories);
  const trajectory  = await scoreTrajectory(input.commits, velocity);

  // Normalize each component against global population
  const [nV, nC, nK, nI, nT] = await Promise.all([
    normalize(velocity,    'velocity'),
    normalize(complexity,  'complexity'),
    normalize(consistency, 'consistency'),
    normalize(impact,      'impact'),
    normalize(trajectory,  'trajectory'),
  ]);

  // Weighted sum → 0 to 10,000 score
  const raw = (nV * 0.20) + (nC * 0.25) + (nK * 0.20)
            + (nI * 0.25) + (nT * 0.10);

  return {
    executionScore: Math.round(raw * 10000),
    velocityScore:    Math.round(nV * 10000),
    complexityScore:  Math.round(nC * 10000),
    consistencyScore: Math.round(nK * 10000),
    impactScore:      Math.round(nI * 10000),
    trajectoryScore:  Math.round(nT * 10000),
    computedAt:       new Date(),
  };
}

Score update cadence: Scores are recomputed whenever the ingestion pipeline detects new commit activity. High-activity developers (100+ commits/month) get near-real-time updates via priority queue. Lower-activity developers are batched.

Normalization strategy: Each sub-score is normalized using a percentile rank against the full developer population using PostgreSQL window functions. This makes the score relative and immune to absolute metric inflation over time.

scoring-engine/normalize.sql 
-- Compute percentile rank for velocity normalization
SELECT
  developer_id,
  velocity_raw,
  PERCENT_RANK() OVER (
    ORDER BY velocity_raw ASC
  ) AS velocity_percentile
FROM developer_scores
WHERE is_bot = FALSE
  AND gaming_flag = FALSE
  AND total_commits > 10;

-- Update execution_score atomically
UPDATE developers
SET
  execution_score  = :score,
  velocity_score   = :v,
  complexity_score = :c,
  global_rank      = RANK() OVER (
                       ORDER BY execution_score DESC
                     ),
  score_updated_at = NOW()
WHERE id = :developer_id;

06 Data Ingestion Pipeline

GitHub ingestion pipeline

The ingestion layer is the foundation. It must be fast, resilient, and immune to GitHub's rate limiting constraints.

ingestion/sync-developer.ts 
export async function syncDeveloper(
  username: string
): Promise<SyncResult> {

  // 1. Fetch GitHub user profile
  const profile = await github.getUser(username);
  await db.developers.upsert({ ...profile });

  // 2. Fetch all repositories (paginated)
  const repos = await github.getUserRepos(username, {
    type: 'owner',
    sort: 'updated',
    per_page: 100,
  });

  // 3. Queue per-repo commit harvesting
  for (const repo of repos) {
    await commitQueue.add('harvest-commits', {
      repoId: repo.id,
      developerId: profile.id,
      since: repo.lastSyncedAt ?? null,
    }, { priority: getPriority(repo.stars) });
  }

  // 4. Fetch language breakdown (GraphQL)
  const languages = await github.getLanguages(repos);
  await db.techStacks.upsertForDeveloper(profile.id, languages);

  // 5. Queue score recalculation
  await scoreQueue.add('recalculate', {
    developerId: profile.id,
  }, { delay: 5000 }); // wait for commits

  return { status: 'queued', repoCount: repos.length };
}

// Rate limit manager — rotates tokens
class RateLimitManager {
  private tokens: string[] = process.env.GITHUB_TOKENS!.split(',');
  private cursor = 0;

  next(): string {
    this.cursor = (this.cursor + 1) % this.tokens.length;
    return this.tokens[this.cursor];
  }

  async handle429(resetAt: Date): Promise<void> {
    const delay = resetAt.getTime() - Date.now() + 1000;
    await sleep(delay); // exponential backoff
  }
}

Delta sync strategy: Only re-fetch commits newer than the last sync timestamp. GitHub ETags are stored per-repository so unchanged responses are not re-processed, reducing API calls by up to 80%.

Token rotation: Multiple GitHub API tokens are rotated round-robin with 5,000 requests/hour each. At 10 tokens, that's 50,000 requests/hour available for ingestion.

Queue priorities: New profile submissions get priority-1 (immediate). Daily delta syncs for active developers get priority-2. Weekly full syncs for inactive devs get priority-5.

Commit deduplication: SHA hashes enforce unique constraints. Re-syncing a repo never produces duplicate commit rows.

07 Anti-Gaming System

Making it tamper-resistant

Any ranking system faces adversarial users. POW implements seven detection signals evaluated per-commit and per-developer to filter non-genuine engineering activity.

🚨 Attack Vector
Empty Commit Spam
Developers creating hundreds of trivial commits (0 additions, 0 deletions, single character changes) to inflate velocity scores. Often scripted via bots with cronjobs.
✓ Detection Method
Minimum Delta Threshold
Commits below a minimum viable delta (additions + deletions < 5) are flagged with gaming_score > 70 and excluded from velocity calculations. Merge commits are also weighted separately.
🚨 Attack Vector
Repetitive File Cycling
Repeatedly modifying the same 1–2 files (e.g. a README counter or a single timestamp file) to simulate active development without contributing real code changes.
✓ Detection Method
File Entropy Analysis
The ratio of unique files touched vs total commit count is tracked per developer. Entropy below a threshold triggers a gaming_flag. Healthy engineers spread changes across many files.
🚨 Attack Vector
Bot Account Activity
Using a bot GitHub account to make commits on behalf of a developer's repositories, or running automated push scripts that generate fake commit histories.
✓ Detection Method
Commit Pattern Analysis
Commits clustered at identical time intervals (e.g. every 60 seconds, every hour on the hour) are flagged. Human commit patterns have natural variance. Perfect regularity is a bot signal.
🚨 Attack Vector
Star Farming
Creating star-exchange networks where developers collectively star each other's repos to inflate the impact signal, particularly in closed communities or automated star-for-star systems.
✓ Detection Method
Star Velocity Anomaly
Repositories gaining more than N stars per day from accounts with low follower counts or accounts created within 30 days of the starring event are weighted down in the impact calculation.

Database scaling strategy

The commits table is the largest growth vector — at 26M developers averaging 2,000 lifetime commits each, that's ~52 billion rows. PostgreSQL declarative partitioning by month keeps individual partition sizes manageable. Old partitions can be moved to cheaper cold storage.

Leaderboard reads are served entirely from Redis-cached materialized snapshots. The live PostgreSQL leaderboard query is only run by background workers, never by user requests.

Score recalculations are stateless workers that read from Postgres and write back scores atomically. They scale horizontally without coordination.

CREATE TABLE commits (
  id           UUID        NOT NULL,
  developer_id UUID        NOT NULL,
  committed_at TIMESTAMPTZ NOT NULL,
  -- ... all other columns

-- Auto-create monthly partitions
CREATE TABLE commits_2025_01
  PARTITION OF commits
  FOR VALUES FROM ('2025-01-01')
              TO ('2025-02-01');

-- Composite index on hot queries
CREATE INDEX idx_commits_dev_time
  ON commits (developer_id, committed_at DESC)
  WHERE gaming_score < 50;
